## 第一部分：RAG 技术详解

### 什么是 RAG？

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将 **信息检索** 与 **大语言模型生成** 相结合的技术架构。

简单来说，RAG 的工作方式就像一个「开卷考试」：
- 传统 LLM 回答问题时，完全依赖训练时记住的知识（「闭卷」）
- RAG 在回答问题前，先从知识库中找到相关资料，再基于这些资料生成回答（「开卷」）

### 为什么需要 RAG？

大语言模型虽然强大，但存在几个核心痛点：

| 痛点 | 说明 | RAG 如何解决 |
|------|------|-------------|
| **知识截止** | 训练数据有时间限制，无法获取最新信息 | 知识库可随时更新 |
| **幻觉问题** | 模型可能生成看似正确实则错误的信息 | 回答基于真实文档，可溯源 |
| **领域盲区** | 通用模型对垂直领域知识不足 | 注入领域专属文档 |
| **数据隐私** | 私有数据不能用于模型训练 | 数据留在本地，仅检索使用 |

### RAG 核心原理

RAG 系统的核心工作流程分为两个阶段：

**阶段一：索引（Indexing）— 离线处理**

```
文档 → 加载解析 → 文本分块 → 向量化(Embedding) → 存入向量数据库
```

1. **文档加载**：将 PDF、Word、TXT 等格式文件转为纯文本
2. **文本分块**：将长文本切分为固定大小的小块（chunk），保留适当重叠
3. **向量化**：通过 Embedding 模型将文本块转为高维向量
4. **存储**：将向量和原文存入向量数据库

**阶段二：检索与生成（Retrieval & Generation）— 在线查询**

```
用户问题 → 向量化 → 相似度检索 → 获取相关文档 → 构造 Prompt → LLM 生成回答
```

1. **问题向量化**：将用户问题用同一 Embedding 模型转为向量
2. **相似度检索**：在向量数据库中检索最相似的 Top-K 个文档块
3. **构造 Prompt**：将检索到的文档块作为上下文，与用户问题组合成提示词
4. **生成回答**：LLM 基于提示词生成最终回答

### 关键参数详解

RAG 系统的效果很大程度上取决于以下三个核心参数的配置。理解它们的作用，才能根据实际场景调优。

#### Chunk Size（文本块大小）

**作用：** 决定文档被切成多大的"碎片"存入向量库。

| Chunk Size | 优点 | 缺点 | 适用场景 |
|-----------|------|------|---------|
| 较小（200~500） | 检索精度高，能定位到具体段落 | 上下文不完整，可能丢失前后文语义 | 问答式检索、FAQ |
| 较大（800~1500） | 每个块包含更完整的信息 | 检索精度下降，可能混入不相关内容 | 长文档分析、报告摘要 |

**举个例子：** 一篇 5000 字的文章，`chunk_size=500` 会切成约 10 个块，`chunk_size=1000` 则约 5 个块。块越小，检索到的内容越"精确"但越"片面"；块越大，内容越"完整"但可能越"模糊"。

**建议：** 中文文档建议 `500~1000`，英文文档建议 `300~800`。如果发现回答不够全面，优先调大这个值。

#### Chunk Overlap（块重叠大小）

**作用：** 相邻两个块之间共享的字符数，用于避免语义在切割边界处断裂。

```
示例：chunk_size=10, chunk_overlap=3

原文：ABCDEFGHIJKLMNOPQRST

块1：ABCDEFGHIJ
块2：   HIJKLMNOPQ    ← 与块1重叠了 HIJ
块3：      OPQRST     ← 与块2重叠了 OPQ
```

**为什么需要重叠？** 假设一句关键信息刚好横跨两个块的边界，没有重叠的话，两个块各拿到半句话，都无法完整表达语义。重叠确保这句话至少在一个块中是完整的。

| Overlap | 效果 |
|---------|------|
| 0 | 无重叠，切割最干净，但可能丢失边界信息 |
| chunk_size 的 10%~20% | 推荐范围，平衡存储开销和语义完整性 |
| 过大（>30%） | 冗余数据太多，增加存储和检索成本，收益递减 |

**建议：** 一般设为 `chunk_size` 的 10%~20%，例如 `chunk_size=500` 时设 `overlap=50~100`。

#### Search Top-K（检索返回数量）

**作用：** 每次提问时，从向量库中检索多少个最相似的文档块作为 LLM 的参考资料。

| Top-K | 优点 | 缺点 |
|-------|------|------|
| 较小（2~3） | 上下文精炼，LLM 不容易被干扰 | 可能遗漏关键信息 |
| 适中（4~6） | 信息覆盖较全面 | 平衡点 |
| 较大（7~10） | 召回尽可能多的相关内容 | 可能引入噪音，LLM 上下文过长导致回答质量下降 |

**关键权衡：** Top-K 越大，LLM 能看到的参考资料越多，回答越全面，但也越容易混入不太相关的内容，反而干扰 LLM 的判断。同时，更多的上下文意味着更高的 Token 消耗和更长的响应时间。

**建议：** 从 `4` 开始尝试。如果发现回答不够全面就调大，如果发现回答啰嗦或偏题就调小。

#### 三个参数如何协同？

```
文档 ──chunk_size──▶ 切成 N 个块 ──overlap──▶ 块之间有冗余保障
                                                    │
用户提问 ──embedding──▶ 向量匹配 ──top_k──▶ 取出 K 个最相关的块
                                                    │
                                              构造 Prompt ──▶ LLM 生成回答
```

一个直觉性的调参策略：
- **回答不全面/遗漏信息** → 先调大 `Top-K`，再考虑调大 `Chunk Size`
- **回答啰嗦/包含无关内容** → 减小 `Top-K` 或减小 `Chunk Size`
- **回答逻辑断裂/上下文不连贯** → 增大 `Chunk Overlap`

## 第二部分：动手实践

### 环境准备

#### 1. 安装 Python 依赖

```bash
pip install -r requirements.txt
```

#### 2. 配置环境变量

编辑项目根目录下的 `config.env` 文件（首次使用需从示例文件复制）：

```bash
cp config.env.example config.env
vim config.env  # 填入你的 API Key
```

> **注意：** `config.env` 包含 API Key 等敏感信息，已被 `.gitignore` 忽略，不会提交到仓库。提交到仓库的是不含密钥的 `config.env.example`。

本项目支持两种模型接入方式：

**方式一：OpenAI 兼容 API（推荐，支持国内外各大模型）**

所有提供 OpenAI 兼容 API 的服务商都可以直接使用，只需修改对应的 `API_BASE_URL` 和 `API_KEY`：

| 服务商 | API_BASE_URL | 模型名称示例 |
|--------|-------------|-------------|
| OpenAI | `https://api.openai.com/v1` | gpt-4o-mini |
| DeepSeek | `https://api.deepseek.com/v1` | deepseek-chat |
| 通义千问 | `https://dashscope.aliyuncs.com/compatible-mode/v1` | qwen-turbo |
| Kimi | `https://api.moonshot.cn/v1` | moonshot-v1-8k |
| 豆包 | `https://ark.cn-beijing.volces.com/api/v3` | doubao-seed-2-0-pro-260215 |

示例 `config.env`（以豆包为例）：

```env
MODEL_PROVIDER=openai_compatible

# LLM 对话模型
LLM_API_KEY=your-api-key
LLM_API_BASE_URL=https://ark.cn-beijing.volces.com/api/v3
LLM_MODEL_NAME=doubao-seed-2-0-pro-260215

# Embedding 向量化模型（可与 LLM 使用不同服务商）
EMBEDDING_API_KEY=your-api-key
EMBEDDING_API_BASE_URL=https://ark.cn-beijing.volces.com/api/v3
EMBEDDING_MODEL_NAME=doubao-embedding-vision-251215
```

> **注意：** LLM 和 Embedding 可以使用不同的服务商。例如 LLM 用 DeepSeek，Embedding 用豆包多模态模型。如果需要图片检索功能，Embedding 模型必须支持多模态（如豆包 `doubao-embedding-vision-251215`）。

**方式二：Ollama（本地部署，完全离线）**

首先安装并启动 Ollama，然后拉取模型：

```bash
ollama pull qwen2
ollama pull nomic-embed-text
```

配置 `config.env`：

```env
MODEL_PROVIDER=ollama
OLLAMA_BASE_URL=http://localhost:11434
```

> **注意：** Ollama 本地模型通常不支持多模态 Embedding，图片检索功能需要使用云端多模态模型。

### 核心模块解析

#### 1. 配置管理 — `config.py`

通过环境变量统一管理所有配置，`MODEL_PROVIDER` 支持两种模式：

- `openai_compatible`：所有兼容 OpenAI API 协议的服务（DeepSeek、千问、Kimi、豆包、OpenAI 等）
- `ollama`：本地 Ollama 部署

#### 2. 文档加载 — `document_loader.py`

支持 4 种文档格式和 6 种图片格式，根据文件后缀自动选择处理方式：

**文本文档：**

| 格式 | 加载方式 |
|------|---------|
| `.txt` | Python 内置文件读取 |
| `.pdf` | PyPDF2 逐页提取文本 |
| `.csv` | Python csv 模块解析 |
| `.docx` | python-docx 提取段落 |

**图片文件：**

| 格式 | 处理方式 |
|------|---------|
| `.png` `.jpg` `.jpeg` `.gif` `.bmp` `.webp` | 转为 base64 Data URL，直接向量化存储（不经过文本分块） |

#### 3. 文本分块 — `text_splitter.py`

使用 LangChain 的 `RecursiveCharacterTextSplitter`，支持中英文分隔符。**仅对文本文档生效，图片不经过分块流程。**

```python
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,       # 每个块的最大字符数
    chunk_overlap=50,     # 相邻块之间的重叠字符数
    separators=["\n\n", "\n", "。", "！", "？", ".", "!", "?", " ", ""],
)
```

**为什么需要重叠？** 重叠部分确保跨块的语义不会丢失，提高检索质量。

#### 4. 向量存储 — `vector_store.py`

使用 ChromaDB 作为向量数据库，零配置、**本地持久化存储**：

- **存储位置**：项目根目录下的 `chroma_db/` 文件夹，运行后自动创建
- **存储内容**：每个文本块的原文、元数据（来源文件名、块索引）和对应的向量
- **持久化**：数据写入磁盘，重启应用后数据仍然存在

核心方法：
- `add_documents()`：将文本文档块向量化后存入数据库
- `add_image_document()`：将图片通过多模态 Embedding 向量化后存入数据库（metadata 中标记 `type: image` 并存储 base64 数据）
- `search()` / `search_with_scores()`：执行相似度检索，返回最相关的文档块（附带距离分数）
- `get_stats()`：查看向量库统计信息（总块数、集合名、存储路径）
- `get_all_documents()`：获取所有存储的文档详情，包括原文、元数据和向量前几维的预览
- `clear()`：清空向量库

#### 5. RAG 链 — `rag_chain.py`

核心的检索增强生成逻辑，流程为：

1. 将用户问题发送给向量库进行相似度检索（带分数）
2. **过滤图片文档**：仅使用文本类型的文档块构建上下文（图片不送入 LLM，因为当前 LLM 是纯文本模型）
3. 构造包含上下文和问题的提示词
4. 调用 LLM 生成回答
5. 返回回答文本、**所有参考来源（包括图片）** 和相似度分数

#### 6. Web 界面 — `app.py`

基于 Streamlit 构建的交互界面，包含两个 Tab：

**Tab 1 — 问答对话**
- 侧边栏：服务商快速选择（豆包/DeepSeek/千问/Kimi 等）、文件上传（支持文档和图片）、参数调节
- 主区域：对话式问答界面
- 每条回答附带可展开的参考来源（文本显示摘要，图片直接展示缩略图）及相似度分数

**Tab 2 — 向量库检查**
- 向量库统计信息（总块数、存储路径）
- 逐条查看存储的文档块：文本显示原文内容，图片显示缩略图；均展示元数据、向量维度和前 8 维预览
- 相似度检索测试：支持**文本检索**（输入文字查询）和**图片检索**（上传图片进行以图搜图/以图搜文）

### 多模态图片支持

本项目支持**图片上传和跨模态检索**，这得益于豆包多模态 Embedding 模型（`doubao-embedding-vision-251215`），它能将文本和图片映射到同一个向量空间。

#### 工作原理

```
传统 RAG（纯文本）：
  文本 → Embedding → 向量 ──┐
  问题 → Embedding → 向量 ──┼── 相似度匹配
                            └──────────────▶ 返回文本块

多模态 RAG（文本 + 图片）：
  文本 → Embedding → 向量 ──┐
  图片 → Embedding → 向量 ──┼── 在同一向量空间中匹配
  问题 → Embedding → 向量 ──┘
                            │
                            ├── 文本块 → 作为上下文送入 LLM 生成回答
                            └── 图片   → 在参考来源中直接展示（不送入 LLM）
```

**核心思路：** 多模态 Embedding 模型能理解图片的语义内容，将其转为与文本同维度的向量。所以当你上传一张猫的图片并提问"猫"，向量检索能找到这张图片，因为它们在语义空间中是接近的。

#### 图片处理流程

1. **上传图片** → 转为 `data:image/{格式};base64,{数据}` 格式
2. **向量化** → 调用豆包多模态 API 的 `image_url` 输入类型获取向量
3. **存储** → 直接写入 ChromaDB（跳过文本分块），metadata 中标记 `type: image` 并存储 base64 数据
4. **检索** → 文本查询或图片查询都能匹配到图片文档
5. **展示** → 检索结果中的图片直接用缩略图展示

#### 图片为什么不送入 LLM？

当前使用的 LLM（`doubao-seed-2-0-pro-260215`）是纯文本对话模型，无法理解图片内容。把 base64 图片数据塞进 prompt 没有意义，只会浪费 Token。因此设计上：
- **Embedding 模型**（多模态）负责理解图片语义并检索
- **LLM**（纯文本）只基于文本上下文生成回答
- 图片作为参考来源附带展示，让用户知道有相关图片被检索到

如果未来切换到支持视觉的 LLM（如 GPT-4o、doubao-vision 等），可以修改 `rag_chain.py` 把图片也送入 LLM 上下文，实现真正的多模态问答。

#### 支持的检索方式

| 检索方式 | 说明 | 位置 |
|---------|------|------|
| 以文搜文 | 输入文字，检索相关文本块 | 问答对话 / 向量库检查 |
| 以文搜图 | 输入文字，检索语义相关的图片 | 问答对话 / 向量库检查 |
| 以图搜图 | 上传图片，检索相似图片 | 向量库检查 Tab |
| 以图搜文 | 上传图片，检索语义相关的文本 | 向量库检查 Tab |

### 运行与验证

#### 启动应用

```bash
streamlit run app.py
```

浏览器会自动打开 `http://localhost:8501`。

#### 验证步骤

1. **上传文档**：在侧边栏点击上传，选择文本文档和/或图片文件
2. **处理文档**：点击「处理文档」按钮，等待文档被分块/向量化（状态栏会显示处理了多少文本块和图片）
3. **验证向量数据**：切换到「向量库检查」Tab
   - 查看总块数是否符合预期（文本块数 + 图片数）
   - 展开任意块，确认文本显示原文内容、图片显示缩略图
   - 查看向量维度（豆包多模态模型为 2560 维）
   - 在「相似度检索测试」中切换文本/图片检索方式，验证跨模态检索效果
4. **提问测试**：切换回「问答对话」Tab，输入问题，例如：
   - "什么是 RAG？"
   - "RAG 有哪些优势？"
   - 与上传图片相关的问题（如上传了猫的图片，可以问"有什么动物？"）
5. **查看来源**：展开回答下方的「参考来源」，确认文本来源有据可依，图片来源能正常展示缩略图

#### 向量数据文件验证

处理文档后，可以在终端查看 ChromaDB 的持久化文件：

```bash
ls -la chroma_db/
```

你会看到 ChromaDB 自动生成的 SQLite 数据库文件和相关索引文件，这就是向量数据的物理存储位置。

## 常见问题

**Q: 支持哪些大模型 API？**
所有兼容 OpenAI API 协议（`/v1/chat/completions`）的服务都可以使用，包括但不限于 OpenAI、DeepSeek、通义千问、Kimi、豆包等。只需在侧边栏选择服务商或手动填写 API Base URL 即可。

**Q: Embedding 模型怎么选？**
如果你的服务商支持 `/v1/embeddings` 接口，可以直接使用。如果需要图片支持，需要使用支持多模态的 Embedding 模型（如豆包的 `doubao-embedding-vision-251215`）。不支持 Embedding 接口的服务商（如部分 Kimi 套餐），建议单独配置 Embedding 服务商。

**Q: 上传图片后提问，为什么 LLM 回答中没有提到图片？**
这是设计如此。当前 LLM 是纯文本模型，无法理解图片内容，所以图片不会作为 LLM 的上下文。图片会出现在回答下方的「参考来源」折叠面板中，以缩略图形式展示。如果需要 LLM 也理解图片，需要切换到支持视觉的 LLM 并修改 `rag_chain.py`。

**Q: 图片检索效果不好怎么办？**
图片检索的质量取决于多模态 Embedding 模型的能力。确保使用的是支持 vision 的模型。另外，图片内容越清晰、主题越明确，检索效果越好。

**Q: 回答不够全面，遗漏了文档中的信息？**
这通常是参数问题。尝试以下调整（在侧边栏操作）：
1. 调大 **Top-K**（如 4 → 6 或 8），让 LLM 看到更多参考资料
2. 调大 **Chunk Size**（如 500 → 1000），让每个块包含更完整的信息
3. 调大 **Chunk Overlap**（如 50 → 100），减少边界处的信息丢失
调整后需要重新点击「处理文档」。

**Q: 向量数据存在哪里？**
存储在项目目录下的 `chroma_db/` 文件夹中，使用 ChromaDB 持久化到本地磁盘。你可以通过 Web UI 的「向量库检查」Tab 查看所有存储的数据详情，包括原文、元数据和向量值。

**Q: Ollama 连接失败？**
确保 Ollama 服务已启动（`ollama serve`），并且已拉取所需模型（`ollama pull qwen2 && ollama pull nomic-embed-text`）。注意：Ollama 本地模型通常不支持多模态 Embedding，图片功能需要使用云端多模态模型。

**Q: 上传大文件很慢？**
可以在侧边栏调大 Chunk 大小（减少总块数），或减小 Top-K 值以加快检索速度。图片文件大小也会影响处理速度，建议上传前适当压缩。

**Q: 如何支持更多文档格式？**
在 `document_loader.py` 中添加新的加载函数，并注册到 `loaders` 字典中即可。图片格式可以在 `IMAGE_EXTENSIONS` 集合中添加。
